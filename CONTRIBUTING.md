# Contributing to Foundation Base

Thank you for contributing to Foundation Base.

Foundation Base is a large, multi-purpose repository. Contributions are easiest to review when they stay focused on one subsystem, include targeted tests, and update the matching documentation.

## Before you begin

### Required tools

For the core Clojure workflow, install:

- Java 21
- Leiningen
- Git

Verify the environment:

```bash
java -version
lein version
git --version
```

Optional subsystems may require Node.js, Python, R, Docker, PostgreSQL, OpenResty, native compilers, browser drivers, or other runtime-specific tools. Do not install every optional dependency unless the area you are changing requires it.

## Set up the repository

```bash
git clone git@github.com:zcaudate-xyz/foundation-base.git
cd foundation-base
lein deps
```

Start a REPL:

```bash
lein repl
```

Install the current project into the local Maven repository:

```bash
lein install
```

## Understand the main areas

| Namespace or path | Purpose |
|---|---|
| `std.*` | Standard libraries and reusable infrastructure |
| `code.*` | Testing, documentation, source management, queries, tooling, and analysis |
| `hara.*` | Language authoring, target models, typing, and code generation |
| `rt.*` | Runtime adapters and execution environments |
| `xt.*` | Portable libraries and cross-target application layers |
| `src-build/` | Walkthroughs, demos, build definitions, and generated projects |
| `src-doc/` | Documentation source |
| `guides/` | Task-oriented Markdown guides |

## Testing

Foundation Base uses `code.test`, not `clojure.test`.

Tests use `fact` and the `=>` assertion form:

```clojure
(ns example.core-test
  (:require [code.test :refer [fact =>]]))

(fact "addition works"
  (+ 1 2) => 3)
```

### Run a targeted namespace

This is the recommended development workflow:

```bash
lein test :only std.lib.collection-test
```

### Run a namespace group

```bash
lein test :with std.lib
```

### Run the complete suite

```bash
lein test
```

The complete suite includes tests for optional runtimes, databases, containers, native tools, and external programs. A minimal workstation may not satisfy every test prerequisite.

When a test depends on an optional program or runtime, use the repository's namespace-level skip conventions rather than allowing setup to fail unexpectedly.

## Documentation

The public documentation site is generated by `code.doc`.

Important locations:

```text
config/publish.edn
config/publish/*.edn
src-doc/documentation/*.clj
public/
```

Generate all configured documentation:

```bash
lein publish
```

Serve generated documentation locally:

```bash
lein serve
```

The repository homepage and documentation homepage are intentionally mirrored:

```text
README.md
src-doc/documentation/main_index.clj
```

When changing the project's description, audience paths, repository map, quick start, or contribution entry points, update both files in the same pull request.

Documentation examples should be one of the following:

- executable `fact` forms when the example is stable and inexpensive;
- non-executing code elements for environment-dependent examples;
- links to source and tests when a full example would be too large.

## Making changes

Prefer a focused branch and a small set of related changes.

A good contribution normally includes:

1. the implementation or documentation change;
2. targeted tests for the affected namespace;
3. updated guides or generated-documentation source when behaviour changes;
4. a clear pull-request description with the commands used for verification.

Avoid combining unrelated refactors, dependency upgrades, generated output, and feature work in one pull request.

## Adding or changing a language target

A language-target contribution should document:

- the grammar or target model being changed;
- supported forms and known limitations;
- whether the target is Stable, Usable, or Experimental;
- how emitted code is verified;
- which native runtime or compiler is required;
- targeted tests for syntax and runtime behaviour where available.

## Adding or changing a runtime

A runtime contribution should document:

- installation prerequisites;
- process, socket, browser, database, container, or embedded-host requirements;
- lifecycle and cleanup behaviour;
- failure and timeout handling;
- how tests skip cleanly when the runtime is unavailable.

## Repository conventions

- Keep namespaces aligned with file paths.
- Prefer targeted test runs during development.
- Do not assume standard `clojure.test` behaviour.
- Keep generated documentation source under `src-doc/` and generated output under `public/`.
- Mark experimental integrations clearly.
- Update source links and examples when namespaces move.
- Do not commit local databases, credentials, runtime caches, or generated secrets.

## Pull requests

In the pull-request description, include:

- what changed;
- why the change is useful;
- the affected namespaces or documentation pages;
- commands used to test or generate the result;
- optional dependencies required for verification;
- known limitations or follow-up work.

For documentation-only changes, state whether `lein publish` was run successfully. For runtime-dependent changes, state which external versions were used.

## Reporting issues

A useful issue report includes:

- the affected namespace, function, guide, or generated page;
- Java, Clojure, Leiningen, and operating-system versions;
- the smallest reproducible command or form;
- expected and actual output;
- relevant runtime or service versions;
- a stack trace or emitted source when applicable.
